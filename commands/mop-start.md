# 项目初始化与流程引导

你是产品开发助手。用户正在启动一个新项目或查看现有项目的状态。

## 任务

### 1. 检查项目状态

检查当前项目目录下是否已存在 `docs/` 目录及相关文档文件：
- `docs/prd.md` — 产品需求文档
- `docs/design-system.md` — 设计规范文档
- `docs/design-system-preview.html` — 设计规范预览页
- `docs/prototype/` — 设计原型（HTML/CSS/SVG）及原型说明文档
- `docs/product-design.md` — 产品功能设计文档
- `docs/tech-design.md` — 技术设计文档
- `docs/tasks.md` — 任务拆解与跟踪
- `docs/devlog.md` — 开发日志
- `docs/test-plan.md` — 测试计划与用例
- `docs/changelog.md` — 变更记录

### 2. 根据状态引导

**如果是全新项目（docs/ 不存在）：**

创建 `docs/` 目录，然后向用户展示以下工作流程：

```
产品开发流程：

1. /mop-req     → 交互式需求收集，输出产品需求文档 (PRD)
   /mop-import  → 或从已有项目/文档导入生成 PRD（可替代 mop-req）
2. /mop-prod    → 基于 PRD 生成产品功能设计文档
3. /mop-ui      → 生成设计规范 + HTML 设计原型，迭代至满意
4. /mop-tech    → 基于产品设计和 UI 原型生成技术设计文档
5. /mop-tasks   → 基于设计文档生成任务拆解
6. /mop-dev     → 选择任务进行开发，完成后自动更新状态
7. /mop-qa      → 创建测试计划，编写测试用例，执行测试
8. /mop-sync    → 需求、设计或技术变更时，同步更新所有文档

建议从 /mop-req 开始进行需求收集；如果已有项目代码或文档，可以用 /mop-import 快速生成 PRD。
```

**如果项目已存在文档：**

读取现有文档，生成项目状态摘要：
- PRD 状态：是否存在，核心内容概要
- 设计规范状态：`docs/design-system.md` 是否存在，主色调和风格概要
- 设计原型状态：`docs/prototype/` 是否存在，包含哪些页面
- 设计文档状态：是否存在，覆盖范围
- 任务进度：总任务数、已完成、进行中、待开始、已延迟、已阻塞
- 测试进度：test-plan 是否存在，通过/失败/待测试数量，未修复 bug 数
- 最近开发记录：devlog 中的最新条目
- 最近变更：changelog 中的最新记录

另外检查 `docs/.mop-checkpoint.md` 是否存在。如果存在，说明上次有技能执行中断，优先提示用户：「检测到上次 /{技能名称} 执行未完成（进度：{当前阶段}）。建议重新运行 `/{技能名称}` 继续完成。」

然后建议用户下一步应该执行的操作。

### 3. 初始化 Git（如果尚未初始化）

如果当前目录不是 git 仓库，提醒用户是否要初始化 git 来进行版本管理。
